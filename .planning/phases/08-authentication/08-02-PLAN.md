<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Login page | Public page, renders for all visitors; sign-in form POSTs to Auth.js endpoint |
| Auth.js callback -> admin pages | OAuth response is validated by Auth.js before session is created |
| Session cookie -> admin layout | Layout reads session for display; proxy.ts + page auth() provide defense-in-depth |
| Browser -> /api/admin/* | Not yet created (Phase 9), but requireAdmin() from Plan 01 is ready |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-09 | Spoofing | Login page CSRF | mitigate | CSRF token fetched from `/api/auth/csrf` and included in sign-in form POST; Auth.js validates token |
| T-08-10 | Elevation of Privilege | Layout-only auth check | mitigate | Auth checked in BOTH admin/layout.tsx (for header display) AND admin/page.tsx (authoritative redirect). proxy.ts provides first layer. Three independent checks per Pitfall 5. |
| T-08-11 | Information Disclosure | Access denied email display | accept | Showing the user's own email is intentional (D-06) -- they need to know which account they used. No other user's data is exposed. |
| T-08-12 | Tampering | signOut server action | mitigate | signOut() calls Auth.js which invalidates the session cookie. If server action form fails (Assumption A1), fallback to form POST with CSRF token. |
| T-08-13 | Spoofing | Direct navigation to /admin bypassing proxy | mitigate | admin/page.tsx calls auth() + checks ADMIN_EMAIL independently of proxy.ts. Even if proxy is bypassed, page-level check redirects unauthorized users. |
</threat_model>
