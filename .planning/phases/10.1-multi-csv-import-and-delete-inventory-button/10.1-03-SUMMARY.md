## Decisions Made

- **Plural error string for the extension guard.** The new validation iterates `files[]` regardless of count, so `"All files must be .csv"` matches the loop semantics even when N=1. Tradeoff: the error string differs from the old single-file `"File must be .csv"`. The 10.1-04 UI will translate the JSON error directly without needing to special-case N=1, so the plural surface is friendlier for the client.
- **Type-narrowing filter on `getAll` result.** `FormData.getAll` returns `FormDataEntryValue[]` (a `File | string` union by spec). The route does `.filter((f): f is File => f instanceof File)` so spurious string parts are dropped silently and the empty-list guard still triggers for completely-empty FormData. This matches the original single-file path's `if (!(file instanceof File)) return 400` semantic.
- **Single-mock pattern in tests.** `parseManaboxCsvContent` is no longer called by the route, so the test file mocks only `parseManaboxCsvFiles`. The library function `parseManaboxCsvContent` remains exported from `src/lib/csv-parser.ts` for any future callers, and its own unit tests in `src/lib/__tests__/csv-parser.test.ts` (Phase 10) continue to cover it directly.
- **Test 4 explicitly asserts the 1-element multi-file array shape.** Even though a single-file upload appears to be a regression-risk-free preserved path, asserting `expect(arg).toHaveLength(1)` and `expect(arg[0].filename).toBe("x.csv")` proves D-01 multi-file uniformity — single-file goes through the same code path as multi-file, just with N=1.
