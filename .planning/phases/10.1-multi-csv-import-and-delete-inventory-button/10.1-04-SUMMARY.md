## Decisions Made

- **`dataTransfer.items.length` for drag-count, not `files.length`.** During `dragOver` browsers expose `items` synchronously (you can read the count) but withhold the actual `File` objects until `drop` fires for security/permission reasons. Reading `items.length` is the correct primitive for "how many files is the user about to drop". `items` is null on some browsers where `files` would be empty too — both code paths handle the `?? 0` case implicitly via the `dragCount > 1` check.
- **Whole-batch rejection (not partial).** If any of the dropped files is non-`.csv`, reject all of them. Same semantics as the 10.1-03 server's "All files must be .csv" 400 response. Partial filtering would be confusing — the user dragged 3 files expecting all 3 to import, partial filtering would silently drop one.
- **`fileLabel` as a module-private helper.** Two render branches (`uploading` and `preview/committing`) need the same display logic; extracting prevents drift if singular/plural copy ever changes.
- **Conditional fragment for filename, not always-render.** `{row.filename && (<>...</>)}` keeps the rendering branch entirely absent when `filename` is undefined (legacy/defensive single-file rows). An always-render with empty string would still output the em-dash separator, polluting the row.
- **Stage union swap, not optional fields.** The plan considered making `filename` optional and adding a sibling `filenames`. Single-field-of-truth (`filenames:string[]` for all paths) is simpler — single-file calls just pass a 1-element array via `files.map(f => f.name)`.
