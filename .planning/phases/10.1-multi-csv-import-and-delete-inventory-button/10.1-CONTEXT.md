# Phase 10.1: Multi-CSV import and delete-inventory button - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend the existing single-file admin CSV import (Phase 10) to accept **multiple CSV files in one upload**, merge them into one combined card list, and commit them as a single atomic full-replace. Add a **destructive "Delete all inventory"** action on `/admin` that wipes the cards table behind a typed-DELETE confirmation.

Both features reuse the Phase 10 atomic-replace contract (`replaceAllCards()` via `db.batch([delete, insert])`); neither introduces additive/upsert semantics.

</domain>

<decisions>
## Implementation Decisions

### Multi-file merge semantics

- **D-01:** Drop zone accepts **multiple** `.csv` files in one upload — drag-drop a selection, or open the file picker with `multiple` attribute.
- **D-02:** All files are parsed independently (`parseManaboxCsvContent` per file), then all `cards[]` arrays are concatenated and passed through the existing `mergeCards()`-style dedup.
- **D-03:** **Cross-file duplicate composite IDs (`setCode-collectorNumber-foil-condition`) sum their quantities** — same behavior as today's within-file dedup. Friend uploads "Binder 1" with 1× Counterspell NM and "Binder 2" with 2× Counterspell NM → merged inventory shows 3×.
- **D-04:** Atomic destructive replace contract preserved verbatim: the merged-and-enriched `cards[]` is posted to the existing `POST /api/admin/import/commit`, which calls `replaceAllCards()`.
- **D-05:** **No additive / upsert mode.** This phase is explicitly merge-then-full-replace, not "add new cards on top of existing inventory." Confirm button label keeps the D-12 safeguard pattern: "Confirm import — replace all {currentTotal} current cards with {toImport} new cards".
- **D-06:** **No hard cap** on file count or combined size. Browser/Vercel limits (~4.5MB request body, 300s `maxDuration` already on `preview`) apply naturally; friend-store usage rarely exceeds 5–10 binders.

### Multi-file UX

- **D-07:** **Pooled progress bar** — single "Fetching prices from Scryfall — X / Y cards" counter spanning all files. Reuses the existing `ProgressBar`. No per-file rows.
- **D-08:** **Skipped rows show source filename** — each entry in the Skipped Rows panel is annotated with which file it came from (e.g., "Row 142 of `Blue Binder.csv`: missing Set code"). Helps the admin trace errors back to the right source file.
- **D-09:** Drop-zone copy updated to reflect multi-file: "Drop one or more Manabox CSVs here…" / "Release to upload" plurals.

### Delete-all inventory

- **D-10:** **Button placement: `/admin` action bar**, near (likely after) Export CSV. Not a separate "danger zone" page. Destructive red styling (`bg-red-600`) matches the import Confirm button.
- **D-11:** **Visibility rule: hidden when inventory is empty (0 cards).** Mirrors the "disabled when toImport===0" pattern from Phase 10 (RESEARCH Pitfall 7) — no point letting the admin "delete nothing".
- **D-12:** **Confirmation pattern: typed "DELETE" modal.** Modal copy: "This will permanently delete all {N} cards. Type DELETE to confirm." Confirm button stays disabled until the input value exactly equals the literal string `DELETE` (case-sensitive, trimmed).
- **D-13:** **Backend: `DELETE /api/admin/inventory` Route Handler** that calls `replaceAllCards([])`. The helper already supports empty input atomically. New route, not a query-string variant of `/commit`, so the intent is explicit in logs and the URL.
- **D-14:** **After delete: stay on `/admin`, render green toast `Deleted all {N} cards` for ~4s** via the existing `sessionStorage.admin-toast` handoff (D-15 from Phase 10). The inventory table re-renders to its empty state from Phase 9.
- **D-15:** **Cart reconciliation already covers it.** The Phase 10 D-13 silent reconciliation effect on `cart-page-client.tsx` already removes items whose IDs vanish from the live card map — applies identically when the entire inventory is wiped, no extra work.

### Branching & deployment

- **D-16:** **All Phase 10.1 work happens on a feature branch** (proposed: `feat/10.1-multi-csv-delete-inventory`). **Do not push to `main` or deploy to production** (`wikos-spellbinder.vercel.app`) until the feature is verified end-to-end on the auto-generated Vercel preview URL.
- **D-17:** STATE.md and ROADMAP.md commits for this phase also live on the feature branch — they merge back together with the code via PR.

### Claude's Discretion

- Specific Tailwind classes for the delete-all modal (overlay, focus trap, backdrop click handling).
- Whether to extract a shared `<ConfirmTypedAction>` component or inline the modal in `/admin`. Probably inline-first; extract on second use.
- Whether the new `DELETE /api/admin/inventory` returns `{ success: true, deleted: N }` or just `{ success: true }`. Probably include `deleted: N` for the toast text.
- Test layout: extend `commit.test.ts` and `preview.test.ts`, or create new spec files? Pick whichever keeps the file under ~250 lines.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `parseManaboxCsvContent(content)` — already returns `{ cards, skippedRows }`; loop over files and concatenate.
- `mergeCards(cards)` private helper in `csv-parser.ts` — already sums quantities by composite ID; works exactly the same on a cross-file concatenated array.
- `replaceAllCards(cards)` — already supports `replaceAllCards([])` for the delete-all path; no schema change needed.
- `Toast` with `variant="success"` (Phase 10-03) — reused for the post-delete toast.
- `sessionStorage.admin-toast` cross-route handoff (Phase 10-03) — reused for the post-delete toast.
- `inventory-table.tsx` already reads + clears `sessionStorage.admin-toast` on mount.
- `cart-page-client.tsx` D-13 silent reconciliation — covers the "delete-all wipes cart items" case for free.

### Established Patterns
- **Confirm label IS the safeguard (D-12 from Phase 10):** the destructive modal text literally states the destructive consequence with the exact count. 10.1 reuses this for the delete-all modal: "This will permanently delete all {N} cards."
- **`requireAdmin()` gate on every admin Route Handler:** new `DELETE /api/admin/inventory` adds it the same way as the existing routes.
- **Atomic full-replace contract via `db.batch()`:** never use `db.transaction()` (drizzle-orm/neon-http doesn't support it).
- **Run-time `nodejs`** for routes that touch the DB or filesystem; `maxDuration=30` for DB-only routes.

### Integration Points
- `/admin/import` page already mounts `ImportClient` — the only UI change for multi-file is internal to that client.
- `/admin` action bar already has Import CSV and Export CSV; the Delete All button slots in next to them.
- The Vercel preview URL for the feature branch is auto-generated; no Vercel project config changes needed.

</code_context>

<specifics>
## Specific Ideas

- Confirmation modal copy: "This will permanently delete all {N} cards. Type **DELETE** to confirm."
- Post-delete toast: "Deleted all {N} cards" (success variant; 4s; reuses existing Toast).
- Drop-zone copy when 1 file dragged: "Drop a Manabox CSV here or click to browse."  When >1 files dragged: "Release to upload N files."  Idle state should accept multi: "Drop one or more Manabox CSVs here, or click to browse."
- Filename annotation in Skipped Rows: prefix with the source filename in subdued text — e.g., `<small>Blue Binder.csv</small> — Row 142: missing Set code`.
- Delete All button label: "Delete all inventory" (NOT "Delete inventory" — the "all" reinforces the destructive scope).
- `DELETE /api/admin/inventory` returns `{ success: true, deleted: N }` so the toast can read the count.

</specifics>

<deferred>
## Deferred Ideas

- Per-file progress bars (we picked pooled). If multi-file imports become routine and the user wants visibility into stuck files, revisit.
- Hard cap on file count (we picked none). Add only if Vercel limits actually bite.
- Audit log of who deleted/imported and when. Would require a table; out of scope for friend store.
- Soft-delete / undo for delete-all. Would require backup/restore infra; deferred — admin can re-import their last CSV instead.
- "Confirm import" → "Confirm full replace (N cards added, M removed)" delta preview. Bigger UI change; not in scope here.

</deferred>
