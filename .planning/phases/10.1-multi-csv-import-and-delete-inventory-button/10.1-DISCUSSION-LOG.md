## Multi-file merge semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Sum quantities | Matches mergeCards() within-file behavior. Friend uploads 'Binder 1' (1x Counterspell NM) + 'Binder 2' (2x Counterspell NM) → inventory shows 3x. Most natural for 'I'm consolidating my collection.' | ✓ |
| Last file wins | Whichever file you drop last overrides earlier. Useful if files represent snapshots in time, but unusual for Manabox exports. | |
| Show as conflict (block until resolved) | Treat duplicate IDs across files as an error — surfaces in skipped-rows panel; admin must dedupe manually before commit. | |
| Hard error (reject the import) | Refuse to even preview if any cross-file duplicate exists. Safest but most annoying. | |

**User's choice:** Sum quantities (Recommended)
**Notes:** Consistent with existing `mergeCards()` within-file behavior — no new mental model.

## Delete-all confirmation pattern

### Q1: How should the admin confirm 'Delete all inventory'?

| Option | Description | Selected |
|--------|-------------|----------|
| Typed DELETE | Modal: 'This will permanently delete all {N} cards. Type DELETE to confirm.' Button stays disabled until exact match. Standard destructive-action pattern (GitHub, Stripe). | ✓ |
| Typed exact card count | Type the current card count (e.g., '847'). Forces the admin to look at the number before confirming. | |
| Click-to-confirm dialog (no typing) | Just two clicks: Delete All → 'Are you sure?' → Yes. Faster but easier to misclick. | |

**User's choice:** Typed DELETE (Recommended)

### Q2: What happens after a successful delete-all?

| Option | Description | Selected |
|--------|-------------|----------|
| Stay on /admin, green toast | Mirrors import-success pattern. Toast: 'Deleted all {N} cards.' Inventory table re-renders showing the empty state from Phase 9. | ✓ |
| Redirect somewhere else | e.g. push to /admin/import to encourage uploading replacement inventory. | |

**User's choice:** Stay on /admin, green toast (Recommended)

## Multi-file UX (progress + skipped rows)

### Q1: When N files are uploaded, how should the enrichment progress display?

| Option | Description | Selected |
|--------|-------------|----------|
| Pooled progress | Single bar: 'Fetching prices from Scryfall — X / Y cards (across N files)'. Reuses existing ProgressBar, simpler to implement and read. | ✓ |
| Per-file progress | List of files, each with its own bar/check. More info but busier UI; no current pattern for it. | |

**User's choice:** Pooled progress (Recommended)

### Q2: In the Skipped Rows panel, should each row show which file it came from?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — show filename per row | 'Row 142 of Blue Binder.csv: missing Set code'. Helps the admin go back to the right file to fix the source. | ✓ |
| No — just row number + reason like today | Keeps the existing UI. Admin has to figure out which file owns each error from context. | |

**User's choice:** Yes — show filename per row (Recommended)

## Limits & guardrails

### Q1: Should there be a hard cap on the number of CSV files uploaded at once?

| Option | Description | Selected |
|--------|-------------|----------|
| No hard cap | Browser/Vercel limits apply naturally (~4.5MB combined body). Friend-store usage rarely exceeds 5–10 binders. Keep the door open. | ✓ |
| Cap at 10 files | Soft ceiling that probably never trips for your usage but prevents pathological '500 files' DoS attempts. | |
| Cap at 5 files | Tighter; matches typical 'few binders' use case. | |

**User's choice:** No hard cap (Recommended)

### Q2: Should the Delete-all button be visible/enabled when inventory is already empty (0 cards)?

| Option | Description | Selected |
|--------|-------------|----------|
| Hide when empty | No point showing a 'delete all' on an empty table; mirrors how Confirm Import disables at toImport===0. | ✓ |
| Show but disable | Always-visible button with tooltip 'Inventory is already empty.' | |
| Always show, always enabled | Clicking with 0 cards = no-op success. Simpler logic, slightly more confusing UX. | |

**User's choice:** Hide when empty (Recommended)
