<objective>
Land the v1.3 binder-aware allocator: rewrite the existing single-CTE `placeCheckoutOrder` in `src/db/orders.ts` to deterministically allocate each buyer line across binder source rows inside ONE SQL CTE chain — `requested → locked_rows (FOR UPDATE OF cards on the aggregated key, ORDER BY binder ASC) → conflicts → can_fulfill → allocations (LEAST(quantity, GREATEST(0, requested - prior_running_supply))) → nonzero_allocations → stock_write → inserted_order → inserted_items` — preserving Phase 11's strict all-or-nothing contract, never overselling under concurrent checkout, and producing one `order_items` row per binder source so admin order detail (Phase 21) renders the operator's exact pick path. Pin the load-bearing concurrency invariant (PITFALLS Pitfall 1: lock by aggregated key, not by chosen rows) with an extension of the Phase 11 concurrent-proof harness covering the multi-binder case (D-07). Keep `StockConflict` shape verbatim while shifting `cardId` to the 4-segment aggregated id and `available` to SUM-across-binders (D-06, D-14). Add the route-handler catch path that translates Postgres CHECK-constraint violations to HTTP 503 (D-08), and emit a new `binderSourceCount` field on the `checkout.order_committed` structured log (D-13).
</objective>

    - The spike artifact is `.planning/phases/18-allocator/18-SPIKE-NOTES.md`. It captures decisions (with one-line justifications and a Postgres-docs link or inline reproduction) for the following questions:

      Q1. **`FOR UPDATE OF cards` syntax in a CTE that joins cards to a VALUES list.**
        - Confirm whether `FOR UPDATE OF cards` (table-name) is the correct form when the SELECT has `FROM cards INNER JOIN requested USING (...)`. Postgres docs (https://www.postgresql.org/docs/current/sql-select.html#SQL-FOR-UPDATE-SHARE) state `FOR UPDATE OF table_name` references the OUTPUT NAME of the table — for a plain `FROM cards`, that's `cards`. Confirm the syntax compiles in a Drizzle `sql\`\`` template by writing a one-line `db.execute(sql\`SELECT id FROM cards INNER JOIN (VALUES ('x')) v(k) ON v.k = cards.id ORDER BY cards.id FOR UPDATE OF cards\`)` against a local Postgres (or accept the Postgres-docs precedent if no local instance available — note in spike-notes which was used).
        - Decision to record: literal SQL fragment to use in `locked_rows` (`FOR UPDATE OF cards` is the recommended form per ARCHITECTURE Q3 line 307).
        - Edge case: if the `requested` VALUES list joins to `cards` via an inner-join derived table, the `OF cards` qualifier prevents Postgres from also trying to lock the (non-table) VALUES rows.

      Q2. **Window function semantics: `ROW_NUMBER()` and `SUM() OVER (... ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)` inside a CTE that has `FOR UPDATE`.**
        - Confirm Postgres allows window functions in the same SELECT as `FOR UPDATE` on a plain table reference. This is supported per Postgres semantics — the `FOR UPDATE` is applied to the rows that flow OUT of the SELECT (i.e., the ones the window functions saw). Sanity-check by reading the Postgres docs note "If a locking clause is present, the table to be locked is the one named directly in the FROM clause" — applies cleanly here.
        - Decision to record: window functions go in the same `SELECT cards.*` projection as the `FOR UPDATE OF cards` clause; no need to split into two CTEs.

      Q3. **`prior_running_supply` arithmetic: derive from `running_supply - l.quantity` in the allocations CTE.**
        - The math: `running_supply` (per ARCHITECTURE Q3) is the cumulative sum INCLUDING the current row. To get the cumulative sum BEFORE the current row (used as the threshold the buyer's request must reach), subtract this row's quantity: `prior_running_supply = l.running_supply - l.quantity`. Then `take_qty = LEAST(l.quantity, GREATEST(0, r.requested_qty - prior_running_supply))`. Validate this arithmetic against the ARCHITECTURE Q3 confidence-MEDIUM fixture matrix BY HAND in the spike notes:
          - `(2, 2, 2)` × 3 → expected `[2, 1, 0]`: row1 qty=2, prior=0, take=min(2, max(0, 3-0))=2; row2 qty=2, prior=2, take=min(2, max(0, 3-2))=1; row3 qty=2, prior=4, take=min(2, max(0, 3-4))=0. ✓
          - `(2, 2, 2)` × 5 → expected `[2, 2, 1]`: row1 take=2; row2 prior=2, take=min(2, 3)=2; row3 prior=4, take=min(2, 1)=1. ✓
          - `(2, 2, 2)` × 6 → expected `[2, 2, 2]`: row3 prior=4, take=min(2, 2)=2. ✓
          - `(2, 2, 2)` × 7 → expected: conflict (total_supply=6 < requested=7); allocations CTE never executes the LEAST/GREATEST branch because `WHERE can_fulfill.ok` is false.
        - Decision to record: the arithmetic is correct; no need to materialize `prior_running_supply` in a separate CTE — inline it in `allocations`.

      Q4. **`db.execute(sql\`...\`)` parameterization with multi-segment aggregated keys.**
        - The current `placeCheckoutOrder` parameterizes the `requested` VALUES with `sql.join(requested.map((item) => sql\`(${item.cardId}::text, ${item.quantity}::integer)\`), sql\`, \`)`. For Phase 18, the aggregated cardId is a 4-segment composite (`set_code-collector_number-finish-condition`). The CTE must split this into 4 named columns so the join to cards works on individual columns (not the concatenated key). Two options:
          - (A) Pass cardId as a single text param, split via `regexp_split_to_array(card_id, '-')` in the CTE. Brittle on binder names with unusual chars (already normalized per Phase 17 D-03 to lowercase + underscore, so hyphen is safe — but the cart key itself is constructed by Phase 20 client code).
          - (B) Aggregate in the JS caller: parse the 4-segment cardId into `{ setCode, collectorNumber, finish, condition }` BEFORE building the VALUES list; pass 4 columns. More explicit, less brittle, no SQL parsing.
        - Decision: **pick option (B)**. Add a `parseAggregatedCardId(cardId: string): { setCode, collectorNumber, finish, condition }` helper at the top of `orders.ts` that splits on `-` and validates the 4-segment shape (throws on malformed input — same defensive-fail as the existing `aggregateCheckoutLines`). The VALUES literal becomes `(${item.setCode}::text, ${item.collectorNumber}::text, ${item.finish}::finish, ${item.condition}::text, ${item.quantity}::integer)`. The `finish` cast to the `finish` enum type is necessary so the `USING (finish)` join matches enum-typed `cards.finish`.
