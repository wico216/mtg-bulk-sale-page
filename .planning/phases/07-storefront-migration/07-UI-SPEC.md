# Phase 7 — UI Design Contract

> Visual and interaction contract for the storefront migration phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase scope:** This is a data-source migration (static JSON to live Postgres). The visual contract is intentionally minimal because the user-facing goal is "zero visible changes." The contract covers only the elements that must change: error state copy, empty state copy updates, and confirmation that the existing design system is preserved exactly.

### Copy rendering rules

Error states render inside the existing page shell (Header visible, centered text in main content area). The pattern is already established in `src/app/page.tsx`:

```
<div className="min-h-screen font-sans bg-white dark:bg-zinc-950 text-zinc-900 dark:text-zinc-100">
  <Header />
  <main className="max-w-7xl mx-auto px-4 py-16 text-center">
    <p className="text-lg text-zinc-600 dark:text-zinc-400">
      {error message here}
    </p>
  </main>
</div>
```

This exact structure must be used for all DB error states on pages. The checkout API route returns JSON, not HTML.

## Visual Parity Checklist

Since the explicit goal is "zero visible changes," the following must be true after migration:

| Check | Requirement |
|-------|-------------|
| Card grid renders identically | Same image, name, set, price, condition, quantity layout |
| Card modal works | Tap/click opens detail modal with oracle text |
| Search/filter/sort works | All filter controls produce the same results |
| Cart persists | localStorage cart survives the migration |
| Checkout flow works | Form submission, email notifications, confirmation page |
| Dark mode works | All pages respect `prefers-color-scheme: dark` |
| Mobile responsive | No layout regressions on mobile viewports |
| Price display | `$X.XX` format preserved (cents-to-dollars conversion in queries.ts) |
