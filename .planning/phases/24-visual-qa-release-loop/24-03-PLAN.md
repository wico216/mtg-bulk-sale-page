# Plan 24-03 — Release Guard + Reusable Playbook

**Phase:** 24 — Visual QA Release Loop
**Risk:** Medium
**Goal:** Make the loop enforceable before release and portable to work/Nova-style UAT.

## Outcome

Atlas can ask “is this gate approved?” and get a clear approved/failed/pending answer before merge/release. The docs explain the repeated loop and map Spellbook concepts to a general product/dev workflow.

## Scope

### In scope

- Add a status/read command or script that can report latest gate review state.
- Fail closed if the gate cannot be found, cannot be read, or is not approved.
- Document the release loop:
  - Issue/feature
  - GSD plan
  - implementation lane
  - Playwright proof
  - QA gate
  - Wiko approve/fail
  - release/smoke
  - regression/test/runbook learning
- Add a short “work transfer” section:
  - Spellbook bug/feature → work ticket/roadmap item
  - Vercel preview → staging environment
  - `/qa/gates` → UAT sign-off page/checklist
  - `admin_audit_log` → ticket/audit trail
  - Playwright proof → QA automation/evidence
- Keep command/docs safe for Telegram/browser-openable links.

### Out of scope

- Making GitHub branch protection depend on the gate.
- Auto-merging after approval.
- Storing sensitive work/Nova data in Spellbook or cloud artifacts.

## Candidate files

- `scripts/check-qa-gate-status.ts` or similar
- `docs/qa-approval-gates.md`
- Maybe `docs/visual-qa-release-loop.md`
- Tests for script logic if factored into a pure helper

## Implementation steps

1. Choose the status source:
   - preferred: app API `GET /api/qa/gates/[runId]/review` when running against preview/prod with QA auth available
   - fallback/local: DB-backed helper in app context for CI/manual environments
2. Add a script or documented command that returns one of:
   - `approved`
   - `failed`
   - `pending`
   - `unreadable`
3. Ensure release guard semantics:
   - approved exits 0
   - failed/pending/unreadable exits non-zero when `--require-approved` is used
4. Update docs with exact loop steps and release report template.
5. Add the work-transfer mapping so the pattern can later be used at Nova without exposing Spellbook internals.
