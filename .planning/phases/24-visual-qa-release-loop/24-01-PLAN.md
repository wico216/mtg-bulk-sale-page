# Plan 24-01 — Gate Packet Generator + Registry Cleanup

**Phase:** 24 — Visual QA Release Loop
**Risk:** Medium-low
**Goal:** Make QA gate packets repeatable and validated so Atlas/agents do not hand-author large `QA_GATE_RUNS` blocks every time.

## Outcome

A developer/agent can define a QA gate packet from a compact structured manifest/factory, validate it in tests, and register it without weakening existing review rules.

## Scope

### In scope

- Split static run data away from helper/type logic if useful.
- Add a typed helper such as `defineQaGateRun(...)` or `createQaGateRunFromManifest(...)`.
- Add validation for required gate fields:
  - `id`, `ticketId`, `title`, `featureArea`, `summary`
  - `proofRun` with command/target/browser/result
  - at least one expected behavior
  - required checklist rows have IDs/labels/expected text
  - artifact URLs are non-empty and kind is valid
- Add a compact reference manifest for the demo/mobile gate.
- Keep existing `getQaGateRun`, `listQaGateRuns`, `normalizeQaChecklist`, and `emptyQaChecklist` behavior stable.
- Update tests around `src/lib/qa-gates.ts` or new registry file.

### Out of scope

- Uploading artifacts to Blob/R2/S3.
- Database-backed gate definitions.
- Auto-generating GitHub Checks.
- Changing review persistence away from `admin_audit_log`.

## Candidate files

- `src/lib/qa-gates.ts`
- `src/lib/qa-gate-runs.ts` or similar new registry module
- `src/lib/__tests__/qa-gates.test.ts`
- `docs/qa-approval-gates.md`

## Implementation steps

1. Inspect current imports of `QA_GATE_RUNS`, `getQaGateRun`, and `listQaGateRuns`.
2. Move only static run data if that reduces file size without churn.
3. Add a tiny gate-definition helper with runtime assertions usable in tests/build.
4. Convert the existing demo gate through the helper.
5. Add tests for:
   - listing still contains existing demo gate
   - invalid manifest/checklist state is rejected or fails validation
   - generated run preserves evidence/checklist/artifact shape
6. Update docs with the new authoring flow.

## Verification

```bash
npm test -- --run src/lib/__tests__/qa-gates.test.ts
npm test -- --run src/app/api/qa/gates/[runId]/review/__tests__/route.test.ts
npx tsc --noEmit
```

## Acceptance criteria

- [ ] Gate metadata authoring is smaller and less error-prone than hand-editing one giant array.
- [ ] Existing QA gate review behavior is unchanged.
- [ ] Required server-side approval/fail-note tests still pass.
- [ ] Docs show the new packet authoring flow.

## Rollback

Revert to the pre-existing hard-coded `QA_GATE_RUNS` array. No data migration involved.
