# Quick Task 260514-7z2 — Buyer Phone (end-to-end)

## Outcome

Operator-grade optional phone field flows end-to-end from the storefront
checkout form, through the API validation + persistence layer, into the
DB as `orders.buyer_phone`, and back out via `getOrderById` to the admin
order detail page as a `tel:` link. Phone is **admin-only context** —
the type system (`PublicOrderData = Omit<OrderData, "items" | "buyerPhone">`)
plus a runtime destructure in the route handler enforce that the
buyer's own `CheckoutResponse` never echoes it back.

## What shipped

### Slice 1 — Schema + migration + types + DB queries (b17ec52)

- `src/db/schema.ts`: nullable `buyerPhone: text("buyer_phone")` on `orders`
  between `buyer_email` and `message`.
- `scripts/migrate-v1.3.x-buyer-phone.ts`: idempotent single-statement
  migration mirroring the v1.3 binder pattern. Pre-flights via
  `information_schema.columns`, supports `--dry-run` + `--help`, prints
  a structured terminal summary. Re-run on an already-migrated schema
  exits 0 with "buyer_phone column already present — no changes".
- `package.json`: new `migrate:phone` and `migrate:phone:dry-run` scripts.
- `src/lib/types.ts`: `CheckoutRequest.buyerPhone?` + `OrderData.buyerPhone?`;
  `PublicOrderData` redefined as `Omit<OrderData, "items" | "buyerPhone"> & { items: PublicOrderItem[] }`.
- `src/db/orders.ts`: `placeCheckoutOrder` threads `buyerPhone` through the
  `inserted_order` INSERT CTE; `getOrderById` SELECT gains `buyer_phone AS "buyerPhone"`;
  `AdminOrderRow` / `PersistedOrder` / `normalizeOrder` all carry the field.

### Slice 2 — API validation + public-response strip + tests (c878688)

- `src/app/api/checkout/route.ts`:
  - `validateCheckoutRequest` adds buyerPhone rules: ≤32 chars + must
    contain ≥1 digit; empty/whitespace/undefined treated as omitted.
    Single error string `"Invalid phone"` for all failure shapes.
  - POST handler threads validated phone into `placeCheckoutOrder` as
    trimmed string or `null`.
  - Public-response strip: `const { buyerPhone: _buyerPhone, ...orderWithoutPhone }`
    destructure before building `publicOrder` — defense-in-depth atop
    the compile-time `Omit`.
- `src/app/api/checkout/__tests__/route.test.ts`: `sampleOrder.buyerPhone: null`
  + 6 new tests (omitted accepted, valid accepted, >32 char rejected,
  no-digit rejected, response stripped, whitespace → null).

## Threat model (STRIDE summary)

| Threat | Disposition | Notes |
|---|---|---|
| Tampering — oversized/malicious phone string | **Mitigated** | 32-char hard cap + drizzle parameterized INSERT; rate-limit bucket before body parse. |
| Information Disclosure — phone leaking into `PublicOrderData` | **Mitigated** | TypeScript-enforced `Omit<…, "buyerPhone">` + runtime destructure in route. Test pins the absence of `buyerPhone` on the response. |
| Information Disclosure — phone in logs | **Mitigated** | No new log site adds buyerPhone to metadata. Existing `scrubErrorMessage` 7+-digit regex covers pg-driver error echo paths. |
| Information Disclosure — admin screen / screenshot | **Accepted** | Admin detail is behind `requireAdmin()`; operator already sees `buyerEmail` in the same card. |
| Repudiation — buyer claims they didn't supply a phone | **Accepted** | orders table is append-only with createdAt timestamps; the phone snapshot lives next to buyerEmail. |
| Spoofing — buyer submits someone else's phone | **Accepted** | Out of scope (same posture as buyerEmail; SMS round-trip verification explicitly deferred). |
| Elevation — clients learn validation rules | **Mitigated** | Single error string `"Invalid phone"` for all rule failures. |

## Out of scope (explicitly deferred)

- SMS sending. Just storing/displaying.
- Buyer accounts / magic-link order history.
- E.164 enforcement / phone formatting.
- Phone search in admin orders.
